# Lantern City — City Seed JSON Schema

## Purpose

This schema formalizes the city seed object used to initialize a new replayable city instance.

It is intentionally structured so that:
- the seed can be generated by an LLM or other generator
- the runtime can validate the result before bootstrapping the city
- the engine can scale city size without changing the backend architecture

## Seed Object

A city seed should be represented as one JSON object with these top-level sections:
- city_identity
- district_configuration
- faction_configuration
- lantern_configuration
- missingness_configuration
- case_configuration
- npc_configuration
- progression_start_state
- tone_and_difficulty

## Validation Rules

### Required top-level keys
The seed must include all of these:
- `schema_version`
- `city_identity`
- `district_configuration`
- `faction_configuration`
- `lantern_configuration`
- `missingness_configuration`
- `case_configuration`
- `npc_configuration`
- `progression_start_state`
- `tone_and_difficulty`

### City identity constraints
- `city_name` must be a non-empty string.
- `dominant_mood` must be an array of 2–4 short strings.
- `baseline_noise_level` must be one of: `low`, `medium`, `high`.

### District constraints
- `district_count` must match the length of `districts`.
- Each district must have a unique `id` and `name`.
- `stability_baseline` must be between 0.0 and 1.0.
- `lantern_state` must be one of: `bright`, `dim`, `flickering`, `extinguished`, `altered`.

### Faction constraints
- `faction_count` must match the length of `factions`.
- Each faction must have a unique `id` and `name`.
- `tension_map` keys should use the format `faction_a|faction_b`.

### Lantern constraints
- `lantern_condition_distribution` must sum to approximately 1.0.
- `lantern_tampering_probability` must be between 0.0 and 1.0.

### Missingness constraints
- `missingness_pressure` must be between 0.0 and 1.0.
- `missingness_risk_level` must be one of: `low`, `medium`, `high`.

### Case constraints
- `starting_case_count` must match the length of `cases`.
- Each case must reference valid district, faction, and NPC IDs.

### NPC constraints
- `tracked_npc_count` must match the number of detailed NPC records or be at least the number of key NPC anchors.
- NPC references must be valid across the seed.

### Progression constraints
- Starting progression values must fit the chosen progress scale.
- Recommended range: 0–100 for each track.

### Tone constraints
- Tone values should be drawn from a controlled vocabulary where possible.
- The seed should remain internally consistent with city identity and mystery style.

## Runtime Rule

The runtime should reject or repair any seed that:
- is missing required keys
- references nonexistent IDs
- uses invalid enum values
- has out-of-range numbers
- creates contradictory city structure

## Design Rule

The seed must be detailed enough to bootstrap the whole run, but compact enough to validate quickly.
It should define the city’s initial shape, not every future event.
